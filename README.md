# DeepNoise — Acoustic Event Classification System

A CNN-based system for classifying industrial machine sounds in real time.
Audio clips are represented as log-Mel spectrograms and fed to a convolutional
neural network trained to distinguish four acoustic event classes. The system
runs both offline (batch evaluation) and online (live microphone capture).

**University ML Project — Part 2 (complete)**

---

## Results

| Model | Test Accuracy | Macro F1 | Notes |
|---|---|---|---|
| SVM baseline | 84.9% | 0.845 | RBF kernel, flattened 22 144-feature vectors |
| CNN — no augmentation | — | — | Severe overfitting: val acc 55% at epoch 2, diverged immediately |
| CNN — with augmentation | **100%** | **1.000** | Best epoch 45/50; val acc 100% from epoch 17 onward |

Augmentation completely resolved overfitting. Without it the model memorized the
training set within 2 epochs and failed to generalize. With it, training and
validation accuracy converged and held through epoch 50.

## Acoustic Classes

| ID | Label | Description |
|---|---|---|
| 0 | `normal_operation` | Steady hum/rumble of healthy equipment |
| 1 | `metallic_impact` | Sharp percussive collisions, drops, hammering |
| 2 | `alarm_tone` | Repetitive synthetic tones, buzzers, warning signals |
| 3 | `silence_ambient` | Background noise / idle state |

## Architecture

```
data/raw/<class>/           raw WAV files (placed manually)
        ↓
preprocess.py               load → mono 22050 Hz → peak-normalize → center-crop/pad to 4 s
        ↓
features.py                 log-Mel spectrogram (128 × 173) → .npy
                            augmented copies for training set only
        ↓
data/spectrograms/          .npy arrays + labels.csv
        ↓
dataset.py                  stratified 70/15/15 split → tf.data pipelines
        ↓
        ├── baseline.py     SVM (RBF, C=10) on flattened 22 144-feature vectors
        │
        └── model.py        CNN: Conv×3 + BN + Pool → GAP → Dense(128) → Dense(4)
                  ↓
              train.py      two runs: --augment false / --augment true
                  ↓
              evaluate.py   accuracy, macro F1, confusion matrix, training curves,
                            error analysis, augmentation comparison
                  ↓
              online.py     real-time mic capture → preprocess → predict → print
```

### CNN Architecture

```
Input (128, 173, 1)
→ Conv2D(32, 3×3)  + BatchNorm + ReLU + MaxPool(2×2)
→ Conv2D(64, 3×3)  + BatchNorm + ReLU + MaxPool(2×2)
→ Conv2D(128, 3×3) + BatchNorm + ReLU
→ GlobalAveragePooling2D
→ Dense(128) + ReLU + Dropout(0.4)
→ Dense(4) + Softmax
```

Loss: `categorical_crossentropy` | Optimizer: `Adam(lr=1e-3)` + `ReduceLROnPlateau`
Primary metric: **macro F1** | Batch: 32 | Max epochs: 50

## Setup

**Python 3.11 is required.** Python 3.12+ may work; Python 3.13+ will not —
TensorFlow has no compatible wheels for those versions.

```bash
# Create and activate virtual environment
python -m venv .venv

# macOS / Linux
source .venv/bin/activate

# Windows (PowerShell)
.venv\Scripts\activate
```

```bash
pip install -r requirements.txt
```

> **Windows note:** Before running any script, set `PYTHONPATH` so that
> `src` is importable as a package:
> ```powershell
> $env:PYTHONPATH = "C:\path\to\DeepNoise"
> ```
> Add this to your shell profile or PyCharm run configuration to avoid
> setting it every session.

> **Windows + TensorFlow:** Native GPU support is unavailable on Windows for
> TensorFlow ≥ 2.11. CPU training works without any extra steps. For GPU
> support, use WSL2.

See `MANUAL_TASKS.md` for dataset download and recording instructions before
running any pipeline step.

## Running the Pipeline

Each step requires the previous step's output. Run them in order.

### 1 — Preprocess raw audio

```bash
python src/preprocess.py --data-dir data/
```

Outputs standardized 22050 Hz mono WAVs to `data/processed/`.

### 2 — Extract spectrograms

```bash
# With augmentation (recommended — generates 5× more training samples)
python src/features.py --data-dir data/ --augment

# Without augmentation
python src/features.py --data-dir data/
```

Outputs `.npy` spectrogram files and `data/spectrograms/labels.csv`.

Verify the output before continuing:
```bash
python -c "import pandas as pd; df=pd.read_csv('data/spectrograms/labels.csv'); print(df.groupby(['class_label','is_augmented']).size())"
```

### 3 — SVM baseline

```bash
python src/baseline.py --data-dir data/ --output-dir results/
```

### 4 — Train the CNN (two required runs)

```bash
python src/train.py --data-dir data/ --augment false --output-dir models/noaug/ --epochs 50
python src/train.py --data-dir data/ --augment true  --output-dir models/aug/  --epochs 50
```

The augmented run uses ~2,400 training samples and takes roughly 20–40 minutes
on CPU depending on hardware.

### 5 — Evaluate

```bash
python src/evaluate.py --model models/aug/cnn_best.keras \
                       --data-dir data/ \
                       --output-dir results/
```

### 6 — Online inference

```bash
# Live microphone mode (Ctrl+C to stop)
python src/online.py --model models/aug/cnn_best.keras

# File fallback (no microphone required)
python src/online.py --model models/aug/cnn_best.keras --file path/to/clip.wav
```

Output format:
```
[14:32:07] Predicted: normal_operation  (confidence: 0.91)
[14:32:11] Predicted: metallic_impact   (confidence: 0.74)
```

### 7 — Run tests

```bash
pytest tests/ -v
```

All tests use synthetic data — no audio files are required.

## Data Augmentation

Three strategies applied to the training set only. Validation and test sets
always receive raw unaugmented spectrograms.

| Strategy | Parameters | Rationale |
|---|---|---|
| Gaussian noise | std = 0.005 | Simulates sensor noise |
| Time shift | ±10% of clip length | Teaches temporal invariance |
| SpecAugment — freq mask | 20 frequency bins zeroed | Robust to partial occlusion |
| SpecAugment — time mask | 30 time frames zeroed | Robust to signal gaps |

Each original training clip produces 4 augmented copies, growing the training
set from ~336 to ~2,260 samples. Results are compared in
`results/augmentation_comparison.md`.

## Key Design Decisions

**Single source of truth.** All constants (`SAMPLE_RATE`, `DURATION`, `N_MELS`,
`INPUT_SHAPE`, `CLASS_LABELS`, etc.) are defined once in `src/config.py` and
imported everywhere. No module redeclares them.

**Shared preprocessing.** `online.py` imports `normalize`, `segment`, and
`compute_melspectrogram` directly from `preprocess.py` and `features.py`.
No preprocessing logic is duplicated, so training and inference apply the same
transformations in the same order.

**Spectrogram normalization.** Each spectrogram is independently scaled to
`[-1, 1]` per sample (min-max) before being fed to the CNN, in both the training
data pipeline and the live inference path. This prevents training/inference
distribution mismatch.

**No data leakage.** Augmented samples are generated from training-split clips
only and tracked with `is_augmented=True` in `labels.csv`. The split logic in
`dataset.py` stratifies on non-augmented samples first, then appends augmented
copies to the training split — they never appear in validation or test sets.

## Multi-Agent Development Workflow

This project was implemented using a structured multi-agent workflow powered by
Claude Code. Four specialized AI agents handled distinct roles across the
development lifecycle, with explicit handoff protocols between them.

### Agents

| Agent | Role | Scope |
|---|---|---|
| **Planner** | Decompose work into task cards, sequence dependencies, route findings | Whole project |
| **Coder** | Implement modules, write unit tests, produce delivery reports | One task at a time |
| **Reviewer** | Spec-compliance audit of a single completed module | Single module |
| **Approver** | Adversarial full-codebase audit — bugs, silent failures, demo risks | Entire codebase |

Each agent has a definition file in `agents/` that specifies its mindset,
procedures, and output format. The Planner invokes the correct agent for each
phase; agents do not self-select.

### Standard Development Loop

```
Planner  →  issues TASK-N card (what to build, acceptance criteria, files, notes)
    ↓
Coder    →  implements exactly the task card — no more, no less
            outputs Delivery Report (files changed, test results, open questions)
    ↓
Reviewer →  audits the single module for spec compliance
            returns PASS or FAIL with findings
    ↓
Planner  →  marks DONE (PASS) or re-queues with findings (FAIL)
```

### Approver Gate

Before any training run or live demo, the Planner invokes the Approver for a
cross-module adversarial audit. The Approver's job is not to be fair — it is to
find every way the system breaks:

- Silent correctness failures (wrong outputs that look plausible)
- Crash paths (division by zero, missing files, hanging I/O)
- Numerical instability (NaN propagation, overflow)
- Data pipeline integrity (leakage, label mismatch, non-deterministic splits)
- Online system failures (device errors, warmup delays, preprocessing mismatches)

```
Planner   →  invokes Approver after all tasks DONE
Approver  →  returns APPROVED / CONDITIONALLY APPROVED / REJECTED
             with Critical / Major / Minor findings
Planner   →  routes Critical findings to Coder
             (REJECTED = hard block on training and demo)
Coder     →  fixes all findings
Approver  →  re-audits; confirms APPROVED or escalates
```

### Why This Approach

The Reviewer and Approver roles are deliberately separated. The Reviewer checks
whether a module matches its spec. The Approver asks a different question:
*given everything that was built, what breaks and when?* Cross-module
inconsistencies, demo-day edge cases, and silent numerical errors are invisible
to a per-module review but surface immediately under adversarial cross-module
analysis.

The Approver's first audit of this project caught a training/inference
distribution mismatch: `online.py` was feeding the CNN raw log-Mel values
(range ≈ [-2.3, +2.9]) while the model was trained on spectrograms normalized
to [-1, 1]. Every live prediction was silently wrong despite perfect offline
metrics. The fix was four lines in `_preprocess`; the re-audit confirmed the
two code paths produce numerically identical output (allclose atol=1e-6).
